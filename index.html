LLM Agents: An Overview

LLM Agents are advanced systems that utilize Large Language Models (LLMs) as their core engine to solve complex tasks requiring reasoning, planning, memory, and the use of external tools. Unlike static applications of LLMs, agents are dynamic and capable of executing multi-step tasks by breaking them into smaller, manageable subtasks.

Core Components of an LLM Agent

Large Language Model (LLM)

The core engine, like GPT-4, providing advanced language understanding, reasoning, and generation.

Prompts

Structured instructions that define the agent's role, capabilities, and task-specific goals.

Memory

Enables agents to handle multi-step tasks by tracking short-term context and retaining long-term information.

Knowledge Retrieval

Allows agents to access real-time or domain-specific information from external sources like vector databases or APIs.

Planning & Reasoning

The ability to decompose complex problems into smaller subtasks and reflect on progress.

Tool Integration

Expands capabilities by using external tools like search engines, calculators, or other APIs.

Tool Integration

A core part of building an agent is giving it access to tools. These are functions the agent can call to interact with the outside world, like searching the web, running calculations, or accessing a database.

The ReAct Framework

A ReAct (Reasoning and Acting) Agent is an advanced framework that enables LLMs to reason through problems while interacting with external tools. It combines two essential capabilities: generating a reasoning trace and executing task-specific actions in a continuous loop.

Key Benefits of ReAct Agents

Improved Accuracy

Breaking problems into smaller steps reduces errors and improves reliability.

Dynamic Interaction

Agents can gather additional data or clarify ambiguities during execution.

Enhanced Problem Solving

Combines reasoning with action to solve real-world, multi-step problems.

Multi-Agent Systems

Multi-agent LLM systems consist of multiple agents that collaborate to solve complex tasks. While single-agent systems are effective for independent cognitive tasks, multi-agent systems shine in scenarios requiring teamwork, extended context management, and dynamic interaction.

LangGraph Overview

LangGraph is a powerful library for building agents that can reliably handle complex tasks. It gives you full control by letting you define an agent's workflow as a graph—a series of connected steps. Before building, it's helpful to understand a few core ideas.

Key Concepts Explained

What is a "Runtime"?

The runtime is the engine that executes your agent's graph. It's the system that runs the nodes in the correct order, manages the flow of data between steps, and handles the overall execution of the workflow you've designed.

What does "Stateful" mean?

A stateful agent has **memory**. It automatically keeps track of the entire history of a task—every message, every tool used, and every result. This "state" allows the agent to make smarter decisions based on past events, not just the most recent input.

What is a "Long-Running" Agent?

Because agents are stateful, they can be paused (e.g., to wait for human approval) and resumed days later, picking up exactly where they left off. This is essential for complex workflows that aren't finished in a single, quick interaction.

Core Benefits

Durable Execution

Build agents that persist through failures and can run for extended periods, resuming from where they left off.

Human-in-the-loop

Incorporate human oversight by inspecting and modifying agent state at any point.

Comprehensive Memory

Create stateful agents with both short-term working memory and long-term memory across sessions.

Debugging with LangSmith

Gain deep visibility into complex agent behavior with visualization tools that trace execution paths.

LangGraph Ecosystem

LangSmith

Trace requests, evaluate outputs, and monitor deployments in one place.

LangSmith Agent Server

Deploy and scale agents effortlessly with a purpose-built deployment platform for long running, stateful workflows.

LangChain

5 Steps to Building an Agent

First, think of your process as a flowchart. Each specific task, like classifying an email or searching documentation, becomes a "node" in your graph.

Key Insight

Breaking a complex process into small, single-responsibility nodes makes your agent resilient, observable, and much easier to debug.

LangGraph Quickstart

LangGraph offers two main ways to build your agent: the **Graph API** and the **Functional API**. This quickstart demonstrates how to build the same calculator agent using both approaches.

Graph API: Best for visualizing complex flows. You explicitly define your agent as a graph of nodes (steps) and edges (connections). This is great for readability and debugging, and it's what we'll visualize in the simulation below.

Functional API: Best for simpler, more linear agents. You define your agent using standard Python control flow (`if`, `while`) within a single decorated function. This can be more concise for straightforward logic.

Run a Local Server

pip install -U "langgraph-cli[inmem]"

Create a new project from a template. This command sets up a starter application for you.

langgraph new path/to/your/app

Navigate into your new app's directory and install the required packages.

cd path/to/your/app
pip install -e .

4. Configure Environment

Create a `.env` file and add your LangSmith API key. You can get one for free from the LangSmith settings page.

LANGSMITH_API_KEY=lsv2...

5. Launch the Development Server

Start the local agent server. This will host your LangGraph application.

langgraph dev

Use the LangSmith Studio UI link from the previous step to visualize, debug, and interact with your running agent.

LangGraph Persistence

The Superpower of Stateful Agents

LangGraph's persistence layer is what elevates a simple workflow to a truly stateful, long-running agent. By automatically saving the agent's memory at every step, it unlocks powerful capabilities that are impossible with stateless services.

Core Concepts: Threads & Checkpoints

What is a Thread?

Think of a thread as a single, continuous conversation or task. Every time you start a new, independent run of your agent, you give it a unique `thread_id`. This allows LangGraph to keep all the history for that specific task completely separate from others.

What is a Checkpoint?

A checkpoint is a snapshot—a "save point"—of your agent's memory (its state) at a specific moment in time. LangGraph automatically creates a new checkpoint after each step in your graph. This is what makes the agent's memory durable.

The Memory Store: Sharing State Across Conversations

While Checkpoints save the history for a single conversation (thread), the **Memory Store** allows an agent to remember information about a user *across multiple conversations*. This is how an agent can remember your name or preferences from one day to the next.

Context Window Management

With short-term memory enabled, long conversations can exceed the LLM’s context window. Here are common solutions to manage the conversation history.

What This Enables

Human-in-the-loop

Long-term Memory

Agents can remember past interactions within the same thread, allowing for rich, context-aware conversations.

Fault Tolerance

Checkpointer Implementation Details

Durable Execution

**Durable execution** allows a workflow to save its progress, so it can be paused and resumed later. This is essential for long-running tasks or workflows that require human approval. If a process crashes, you can restart it from the last saved step without re-running everything.

Already Enabled!
If you're using a checkpointer, you already have durable execution enabled. The key is to design your workflow correctly to take advantage of it.

Interactive Simulation: Consistent Replay

This simulation shows why you must wrap operations with side-effects (like API calls) in a `@task`. When a workflow resumes, LangGraph **replays** the steps. Without `@task`, the API call runs again, leading to inconsistent results. With `@task`, the result from the first run is loaded from the checkpoint, ensuring consistency.

Problem: Inconsistent Replay

import requests

def get_time(state: dict):
    # This external API call is a side-effect
    # and is not wrapped in a @task.
    response = requests.get(
        "https://worldtimeapi.org/api/ip"
    )
    # On resume, this function runs again,
    # fetching a NEW time.
    return {"time": response.json()["datetime"]}

Durability Modes

You can choose how often to save progress, balancing performance and safety.

sync

Saves state before each step. Safest, but slowest.

graph.stream(
    {"input": "test"},
    stream_mode="updates",
    config={"durability": "sync"}
)

Streaming in LangGraph

LangGraph provides a powerful streaming system to surface real-time updates. By displaying output progressively, streaming significantly improves user experience, especially when dealing with LLM latency.

Stream Modes Explained

ModeDescription
`values`Streams the full value of the state after each step of the graph.
`updates`Streams only the updates to the state after each step of the graph.
`messages`Streams token-by-token output from LLMs within the graph.
`custom`Streams any custom data that you manually emit from within your nodes.
`debug`Streams as much information as possible for deep debugging.

Human-in-the-loop: Interrupts

Interrupts allow you to pause graph execution at specific points and wait for external input before continuing. This enables powerful human-in-the-loop patterns. When an interrupt is triggered, LangGraph saves the graph's state and waits indefinitely until you resume execution.

  • Checkpointing keeps your place: The checkpointer writes the exact graph state so you can resume later.
  • `thread_id` is your pointer: This tells the checkpointer which conversation state to load.
  • Interrupt payloads are returned: The data you pass to `interrupt()` is returned to your application, so you know what the graph is waiting for.

Model Context Protocol (MCP)

Model Context Protocol (MCP) is an open protocol that standardizes how applications provide tools and context to LLMs. Think of it as a universal bridge between your AI agent and external tools, like weather APIs, calculators, or internal business systems. Instead of being hardcoded, tools are exposed as independent, plug-and-play services.

MCP Server

Exposes functions (tools) that your agent can call.

MCP Client

Connects your LangGraph agent to the available tools.

Why is MCP Powerful?
MCP allows for a plug-and-play architecture. You can run your tools as separate microservices, and your agent can dynamically discover and use them as needed. This makes your system modular, scalable, and secure.

MCP Agent Simulation

Code Setup

from mcp.server.fastmcp import FastMCP

mcp = FastMCP("StringTools")

@mcp.tool()
def reverse_string(text: str) -> str:
    """Reverses the given string."""
    return text[::-1]

@mcp.tool()
def count_words(text: str) -> int:
    """Counts the number of words in a sentence."""
    return len(text.split())

if __name__ == "__main__":
    mcp.run(transport="stdio")

Transports

MCP supports different mechanisms for client-server communication. The two main types are `stdio` for local tools and `http` for remote services.

STDIO Transport

The client launches the server as a local subprocess and communicates via its standard input/output. This is best for local tools and simple setups.

client = MultiServerMCPClient({
    "math": {
        "transport": "stdio",
        "command": "python",
        "args": ["/path/to/math_server.py"],
    }
})
